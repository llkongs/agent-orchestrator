# state/ -- Pipeline Runtime State

Runtime state persistence for pipeline executions. This directory does not yet exist on disk but is referenced throughout the architecture and code.

## Planned Structure

```
state/
  active/          -- Currently running pipelines
    {run-id}/
      pipeline.state.yaml   -- Pipeline + slot statuses, timestamps, gate results
      event-log.yaml        -- Append-only compliance observer event log
      audit/                -- Compliance auditor output (write-only for COMP-001)
  archive/         -- Completed pipeline runs (moved from active/ on completion)
    {run-id}/      -- Same structure as active/
```

## State File Format

Pipeline state YAML contains:
- **pipeline_id**: Template ID used
- **run_id**: Unique execution identifier
- **status**: PipelineStatus enum (loaded/validated/running/paused/completed/failed/aborted/auditing)
- **slots**: Dict of slot_id -> SlotState, each containing:
  - **status**: SlotStatus enum (pending/blocked/ready/pre_check/in_progress/post_check/completed/failed/skipped/retrying)
  - **assigned_agent**: Agent ID filling this slot
  - **started_at / completed_at**: ISO timestamps
  - **gate_results**: List of GateCheckResult for pre/post conditions
  - **metrics**: DeterministicMetrics (test_count, coverage, checksum, etc.)
- **created_at / updated_at**: ISO timestamps

## Slot State Machine

Valid transitions defined in `engineer/src/pipeline/state.py`:
```
PENDING -> BLOCKED | READY | PRE_CHECK | IN_PROGRESS | SKIPPED | FAILED
BLOCKED -> READY | PRE_CHECK | IN_PROGRESS | SKIPPED | FAILED
READY -> PRE_CHECK | IN_PROGRESS | SKIPPED | FAILED
PRE_CHECK -> IN_PROGRESS | FAILED
IN_PROGRESS -> POST_CHECK | COMPLETED | FAILED | RETRYING
POST_CHECK -> COMPLETED | FAILED
RETRYING -> IN_PROGRESS | FAILED
COMPLETED -> (terminal)
FAILED -> RETRYING | (terminal)
SKIPPED -> (terminal)
```

## Persistence Rules

- State is persisted atomically (write to temp file, then rename) by `PipelineStateTracker`
- Event logs are append-only (ComplianceObserver appends YAML documents separated by `---`)
- State files use `yaml.safe_dump()` for serialization
- Checksums are computed with SHA-256 for integrity verification
