# specs/pipelines/templates/ -- Pipeline Template Definitions

All templates are v2.0.0 slot-based format. Each defines a DAG of typed slots with explicit data_flow edges, parameters, and gate conditions.

## Templates

### standard-feature.yaml
- **Slots:** 5 (designer -> implementer -> reviewer -> approver -> deployer)
- **Parameters:** 3 (feature_name, phase_id, target_module)
- **Use case:** Baseline development pipeline. Architect designs -> Engineer implements with DELIVERY.yaml -> QA reviews with REVIEW.yaml -> CEO approves -> Deploy. Most common pipeline.

### research-task.yaml
- **Slots:** 3 (researcher -> architect-review -> approver)
- **Parameters:** 2 (research_topic, research_brief)
- **Use case:** Lightweight pipeline for investigation tasks. No code produced -- output is research reports and recommendations. Researcher studies topic, Architect reviews feasibility, CEO decides Go/No-Go.

### hotfix.yaml
- **Slots:** 4 (implementer -> reviewer -> approver -> deployer)
- **Parameters:** 4 (bug_id, bug_description, affected_module, phase_id)
- **Use case:** Emergency bug fix. Skips design phase. Engineer fixes directly -> QA verifies no regression -> CEO approves -> Deploy. Only for small, well-scoped fixes.

### quant-strategy.yaml
- **Slots:** 6 (designer -> researcher + designer [parallel] -> implementer -> reviewer -> approver)
- **Parameters:** 3 (strategy_name, target_symbol, phase_id)
- **Use case:** Quantitative trading strategy development. Architect scopes constraints, then signal research and strategy design proceed in parallel, Engineer implements, QA verifies end-to-end.

### security-hardening.yaml
- **Slots:** 6 (auditor -> designer -> implementer -> reviewer -> auditor [re-audit] -> approver)
- **Parameters:** 2 (audit_scope, phase_id)
- **Use case:** Security audit + remediation. Two-pass audit: Security auditor identifies vulnerabilities -> Architect designs fix -> Engineer implements -> QA reviews -> Security re-audits to confirm resolution -> CEO approves.

### compliance-audit.yaml
- **Slots:** 3 (compliance-auditor with pre-check gates)
- **Parameters:** 5 (target_pipeline_id, target_pipeline_run_id, state_dir, audit_scope, evidence_paths)
- **Use case:** Post-pipeline compliance audit. Reviews entire execution record of a completed pipeline run. Independent of delivery chain -- can audit all roles including QA and PMO. Produces Process Audit Report with metrics and findings.
