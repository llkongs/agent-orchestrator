# specs/pipelines/slot-types/ -- SlotType Definitions

Each SlotType defines an interface contract: input_schema, output_schema, required_capabilities, and constraints. Agents are matched to slots by capability intersection.

## Slot Types

### designer.yaml
- **Category:** architecture
- **Required capabilities:** system_design, interface_definition, technical_documentation
- **Purpose:** Produces design documents and module interfaces. Does NOT write implementation code. Must reference integration-contract.md and define typed signatures.

### researcher.yaml
- **Category:** research
- **Required capabilities:** web_search, technical_analysis, structured_report_writing
- **Purpose:** Investigates topics and produces structured research reports with findings, confidence levels, and cited sources. Does NOT write implementation code.

### implementer.yaml
- **Category:** engineering
- **Required capabilities:** python_development, test_writing, delivery_protocol
- **Purpose:** Writes production code and tests per design spec. Follows integration contract exactly. Must produce DELIVERY.yaml, achieve >=85% coverage, 0 test failures.

### reviewer.yaml
- **Category:** quality
- **Required capabilities:** independent_testing, code_review, delivery_protocol, cross_validation
- **Purpose:** Independently verifies deliveries. Runs tests independently, cross-validates metrics against DELIVERY.yaml claims, writes boundary tests, produces REVIEW.yaml. Never trusts producer's self-reported numbers.

### approver.yaml
- **Category:** governance
- **Required capabilities:** decision_making
- **Purpose:** Makes Go/No-Go decisions (approved/rejected/deferred). Typically CEO/Team Lead for high-stakes, or PMO for lower-stakes. Must check for P0/P1 blocking issues before approving.

### deployer.yaml
- **Category:** operations
- **Required capabilities:** deployment, ssh_operations, service_management
- **Purpose:** Deploys approved code. Handles PVE snapshots, file transfer, service restart, smoke tests. Must be able to rollback on failure.

### auditor.yaml
- **Category:** security
- **Required capabilities:** security_audit, owasp_review, infrastructure_review
- **Purpose:** Security audits of code, infrastructure, and configuration. Produces structured findings with severity ratings (CRITICAL/HIGH/MEDIUM/LOW). Can perform re-audits to verify resolution.

### compliance-auditor.yaml
- **Category:** governance
- **Required capabilities:** process_audit, artifact_validation, noncompliance_tracking, trend_analysis, objective_evaluation, structured_report_writing
- **Purpose:** Post-hoc process compliance auditing (CMMI PPQA). Reviews gate execution, artifact completeness, protocol adherence. Independent of delivery chain. No veto power -- observe, measure, report only. Distinct from security auditor.
